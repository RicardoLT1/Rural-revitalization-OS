# Xiangyun OS V1.2 Roadmap

V1.2 moves Xiangyun OS from a contest demo toward a maintainable, extensible, and deliverable product baseline.

## Principles

```text
Freeze first, then clean.
Secure first, then decouple.
Close business loops first, then add AI.
Each phase must pass its gate before the next phase starts.
```

## Phase 0: Freeze Current Demo

Goal: keep a stable rollback point before cleanup and refactoring.

Deliverables:

- `archive/current-demo-backup` branch
- `v1.1-current-demo` tag
- Current demo validation record

Gate:

- Do not delete legacy code before Phase 0 is complete.
- Do not clean Mini Program JavaScript artifacts before Phase 0 is complete.
- Do not start internal signing, RabbitMQ, or AI refactoring before Phase 0 is complete.

## Phase 1: Establish Clean Baseline

Goal: keep only the current distributed architecture and TypeScript Mini Program source in the main development line.

Deliverables:

- Legacy `backend/src` removed from the main branch
- Mini Program JavaScript and sourcemap artifacts cleaned
- `.gitignore` updated for local and generated files
- README used as the main documentation entry
- `v1.1-baseline-passed` tag after tests pass

Gate:

- Do not enter Phase 2 before Phase 1 is complete.
- Do not start RabbitMQ or AI work before Phase 1 is complete.
- Do not add new service-to-service security changes before the baseline is clean.

## Phase 2: Internal Security Boundary

Goal: prevent downstream services from trusting forged user headers or direct bypass calls.

Deliverables:

- Internal signature utilities in `xiangyun-common`
- Gateway internal request signing
- Gateway cleanup for externally supplied trusted headers
- Feign internal request signing for service-to-service calls
- Redis-backed nonce replay protection
- `InternalFilter` in Operation and Analysis
- Query-string-aware signatures
- Service allowlists, exclude paths, environment switches, TraceId propagation, and security tests

Gate:

- Do not deploy services to real or staging-like public environments before Phase 2 is complete.
- Do not expose Operation or Analysis directly.
- Do not treat `X-User-Id` or `X-User-Role` as the only identity proof.
- Do not make Feign service calls to protected services without `InternalFeignRequestInterceptor`.
- Do not treat request bodies as fully signed until Stage 2.1 adds body hash verification.

## Phase 3: Formal Web Admin

Goal: replace the static demo Web page with a maintainable Vue 3 admin application.

Deliverables:

- `web-admin/` project
- Login, role menu, routing, Axios interceptors, and environment config
- Approval flow, resource archive pages, dashboard, and weekly report draft entry

Gate:

- Do not retire the static Web demo before Phase 3 is complete.
- Do not fully migrate the official approval workflow to `web-admin` before acceptance.
- Keep the static Web page as a fallback until the new admin flow is verified.

## Phase 4: Outbox + RabbitMQ

Goal: reliably turn business changes into analysis, cache, and reporting events.

Deliverables:

- Operation `msg_outbox`
- Analysis `msg_inbox`
- RabbitMQ exchange and queue
- Publisher Confirm handling
- afterCommit wake-up and scheduled fallback scan
- Idempotent Analysis consumer and retry handling

Gate:

- Do not make RabbitMQ the only Analysis data source before Phase 4 is complete.
- Do not remove Feign query and fallback paths before the event pipeline is verified.
- Treat RabbitMQ as an enhancement path before acceptance, not the only path.

## Phase 5: AI Copilot Service

Goal: introduce low-risk AI draft generation without entering the core decision path.

Deliverables:

- `xiangyun-ai-service`
- Async AI task model
- `WeeklyReportAgent`
- Weekly report JSON draft API
- Web draft review and manual confirmation entry

Gate:

- AI must not write to business databases directly.
- AI must not auto-submit approvals, ownership confirmation, contracts, or project stage changes.
- AI output is only a draft or suggestion until a staff member confirms it.
- AI failure must not affect the core resource, approval, or dashboard flows.

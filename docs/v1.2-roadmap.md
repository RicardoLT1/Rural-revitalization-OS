# Xiangyun OS V1.2 Roadmap

V1.2 moves Xiangyun OS from a contest demo toward a maintainable, extensible, and deliverable product baseline.

## Principles

```text
Freeze first, then clean.
Secure first, then decouple.
Close business loops first, then add AI.
Each phase must pass its gate before the next phase starts.
```

## Phase 0: Freeze Current Demo

Goal: keep a stable rollback point before cleanup and refactoring.

Deliverables:

- `archive/current-demo-backup` branch
- `v1.1-current-demo` tag
- Current demo validation record

Gate:

- Do not delete legacy code before Phase 0 is complete.
- Do not clean Mini Program JavaScript artifacts before Phase 0 is complete.
- Do not start internal signing, RabbitMQ, or AI refactoring before Phase 0 is complete.

## Phase 1: Establish Clean Baseline

Goal: keep only the current distributed architecture and TypeScript Mini Program source in the main development line.

Deliverables:

- Legacy `backend/src` removed from the main branch
- Mini Program JavaScript and sourcemap artifacts cleaned
- `.gitignore` updated for local and generated files
- README used as the main documentation entry
- `v1.1-baseline-passed` tag after tests pass

Gate:

- Do not enter Phase 2 before Phase 1 is complete.
- Do not start RabbitMQ or AI work before Phase 1 is complete.
- Do not add new service-to-service security changes before the baseline is clean.

## Phase 2: Internal Security Boundary

Goal: prevent downstream services from trusting forged user headers or direct bypass calls.

Deliverables:

- Internal signature utilities in `xiangyun-common`
- Gateway internal request signing
- Gateway cleanup for externally supplied trusted headers
- Feign internal request signing for service-to-service calls
- Redis-backed nonce replay protection
- `InternalFilter` in Operation and Analysis
- Query-string-aware signatures
- Service allowlists, exclude paths, environment switches, TraceId propagation, and security tests

Gate:

- Do not deploy services to real or staging-like public environments before Phase 2 is complete.
- Do not expose Operation or Analysis directly.
- Do not treat `X-User-Id` or `X-User-Role` as the only identity proof.
- Do not make Feign service calls to protected services without `InternalFeignRequestInterceptor`.
- Do not treat request bodies as fully signed until Stage 2.1 adds body hash verification.

Completion record:

- Stage 2 development and unit tests were completed on `feature/internal-signature`.
- Runtime integration verified Gateway -> Operation, Gateway -> Analysis, and Analysis -> Operation Feign signing.
- Direct Operation / Analysis business access returns 403 without a valid internal signature.
- `/actuator/health` remains accessible for all four services.
- Final runtime-verified tag: `v1.2-internal-auth-passed-final`.

## Phase 3: Formal Web Admin

Goal: replace the static demo Web page with a maintainable Vue 3 admin application.

Deliverables:

- `web-admin/` project
- Login, role menu, routing, Axios interceptors, and environment config
- Approval flow, resource archive pages, dashboard, and weekly report draft entry

Progress record:

- Phase 3 development started on `feature/web-admin` after Phase 2 was merged and retested on `main`.
- Batch 1 establishes the Vue 3 + TypeScript + Vite project, Pinia session state, Axios authentication handling, role-aware routing, environment proxy, responsive login, and the formal admin shell.
- Production build and desktop/mobile rendering checks pass for Batch 1.
- Batch 2 connects the live dashboard, approval workbench, approval history, resource filters, resource detail drawer, and ADMIN-only resource publish/offline controls.
- Batch 2 runtime checks pass with both STAFF and ADMIN accounts. STAFF cannot see resource management controls; ADMIN can see them. Desktop and mobile pages have no document-level horizontal overflow.
- Batch 3 adds Flyway-managed weekly report persistence, staff-confirmed weekly report editing, ADMIN user administration, and resource create/edit forms.
- Final acceptance covers STAFF and ADMIN routing, live dashboard data, approval lists and decisions, resource archives and role controls, weekly report write/read, user administration, production builds, and desktop/mobile rendering.
- Phase 3 functional acceptance is complete on `feature/web-admin`. The static `web/` fallback remains in the repository until the branch is merged and the retirement decision is recorded.
- Web Admin productization closure is complete on `feature/ui-polish` at commit `74c8c85`, with supporting resource activity API commit `2358e39`.
- Approval rows now open a full context drawer with workflow history and operation logs; processed workflows are read-only.
- Resource archives support independent detail routes, related approval drilldown, operation timelines, photo preview, and real-coordinate map entry.
- Dashboard KPI cards, resource distribution, risks, intelligent suggestions, notifications, search, user menu, and CSV export all produce explicit results.
- Loading, empty, failure, retry, disabled, hover, focus, and transition states share a consistent interaction and visual system.
- Production build, 68 backend tests, and browser regression passed before creating tag `v1.2-web-admin-productized`.

Known V1.2 boundaries:

- Resource map is a lightweight village distribution view based on real coordinates, not a full GIS SDK integration.
- Field photos support display and preview; upload, classification, deletion, metadata, and cover management are not included.
- Global search uses a frontend aggregation service with debounce and recent visits; a federated `/api/search` endpoint remains future work.
- Notifications are derived from live workflow state; persistent read state and a dedicated notification table remain future work.
- Resource activity combines workflow and operation-log events. A dedicated append-only resource activity model can be introduced when audit depth requires it.
- Browser regression is currently manual; a minimal Playwright E2E suite is planned as an engineering follow-up.

Gate:

- Do not retire the static Web demo before Phase 3 is complete.
- Do not fully migrate the official approval workflow to `web-admin` before acceptance.
- Keep the static Web page as a fallback until the new admin flow is verified.

Next experience phase:

- Phase 5.3 focuses on Mini Program visual consistency, resource discovery, application progress, supplement-material guidance, and unified loading/empty/error states.
- AI Copilot remains after dual-client experience alignment and must stay outside approval decisions.

## Phase 4: Outbox + RabbitMQ

Goal: reliably turn business changes into analysis, cache, and reporting events.

Deliverables:

- Operation `msg_outbox`
- Analysis `msg_inbox`
- RabbitMQ exchange and queue
- Publisher Confirm handling
- afterCommit wake-up and scheduled fallback scan
- Idempotent Analysis consumer and retry handling

Completion record:

- RabbitMQ is included in `docker-compose.demo.yml` with a management UI and health check.
- Operation writes `workflow.status.changed` to `msg_outbox` in the same approval transaction.
- The publisher wakes after commit, scans on a fixed fallback schedule, requires publisher confirm, handles unroutable returns, and records retry state.
- Analysis stores unique events in `msg_inbox`, maintains an incremental event aggregate, and invalidates dashboard caches after processing.
- Runtime integration verified `APPROVED -> Outbox SENT -> Inbox PROCESSED`; dedicated integration data was removed afterward.
- Feign dashboard queries and stale-cache fallback remain active. RabbitMQ is an enhancement path, not the only Analysis data source.

Gate:

- Do not make RabbitMQ the only Analysis data source before Phase 4 is complete.
- Do not remove Feign query and fallback paths before the event pipeline is verified.
- Treat RabbitMQ as an enhancement path before acceptance, not the only path.

## Phase 5: AI Copilot Service

Goal: introduce low-risk AI draft generation without entering the core decision path.

Deliverables:

- `xiangyun-ai-service`
- Async AI task model
- `WeeklyReportAgent`
- Weekly report JSON draft API
- Web draft review and manual confirmation entry

Gate:

- AI must not write to business databases directly.
- AI must not auto-submit approvals, ownership confirmation, contracts, or project stage changes.
- AI output is only a draft or suggestion until a staff member confirms it.
- AI failure must not affect the core resource, approval, or dashboard flows.

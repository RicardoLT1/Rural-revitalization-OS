# 第五阶段：配置与交付规范

## 阶段目标

第五阶段解决“项目交给别人后能否稳定启动、验收、演示”的问题。重点不是增加业务功能，而是把配置边界、启动顺序、验收清单和敏感信息规则整理清楚。

本阶段保持轻量，不引入配置中心复杂治理、CI/CD 平台或容器编排。

## 已实现范围

### 1. 配置样例

根目录新增：

```text
.env.example
```

用途：

- 列出后端和基础设施使用的环境变量。
- 给出本地/演示环境默认值说明。
- 明确真实密钥应保存在本地 `.env`，不提交仓库。

说明：当前 PowerShell 启动脚本不会自动加载 `.env`。需要覆盖默认值时，在终端会话中设置环境变量后再启动服务即可。

小程序新增：

```text
miniprogram/config/env.example.ts
```

用途：

- 说明小程序 API 入口、Mock/API 切换、演示账号和模块开关。
- 保持与 `miniprogram/config/env.ts` 的结构一致，方便交付时核对。

### 2. 敏感配置忽略规则

`.gitignore` 已增加：

```text
.env
.env.*
!.env.example
```

这样可以提交 `.env.example`，但不会误提交真实 `.env` 或带环境名的密钥文件。

### 3. 交付检查清单

新增：

```text
docs/delivery-checklist.md
```

覆盖：

- 配置检查。
- 后端启动检查。
- 自动化测试。
- 演示账号。

- 排障信息采集。

### 4. 文档入口

README 已加入第五阶段文档和交付检查清单，形成从架构、业务、可靠性、运维到交付的完整阅读顺序。

## 配置分层

### 本地开发

- 使用 `docker-compose.demo.yml` 启动 MySQL、Redis、Nacos。
- 后端默认连接 `127.0.0.1` 端口。
- 小程序 `baseURL` 指向 `http://127.0.0.1:8080/api`。
- 可以使用演示账号。

### 演示环境

- 允许使用演示账号和演示数据。
- 可以保留较短缓存时间，方便看板演示。
- `JWT_SECRET` 可以使用演示强度密钥，但应不同于仓库默认值。

### 生产预备环境

- 必须替换 `JWT_SECRET`、数据库密码、Redis 密码。
- 不应开启自动登录。
- 小程序 `baseURL` 应指向正式 HTTPS 网关域名。
- 健康检查只暴露 `health/info`，不开放详细运行指标。

## 验收标准

1. `.env.example` 覆盖后端当前使用的环境变量。
2. `.gitignore` 能避免真实 `.env` 文件被提交。
3. 小程序有可对照的环境配置样例。
4. 交付检查清单覆盖启动、测试、演示和排障。
5. 后端 `mvn test` 通过。

## 当前边界

本阶段暂不做：

- CI/CD 自动发布。
- Kubernetes 部署。
- 多环境配置中心治理。
- 密钥管理平台。
- 线上域名和证书配置。

这些能力需要明确部署环境后再做，否则容易变成空转的复杂度。

## 下一阶段建议

第六阶段建议进入“最终验收与演示打磨”：

- 按交付检查清单完整跑一遍。
- 录制或整理演示路径。
- 检查小程序关键页面文字、空状态和异常状态。
- 再做一次全量测试和文档一致性审查。

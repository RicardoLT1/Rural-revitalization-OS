# 第三阶段：可靠性与可观测性基线

## 阶段目标

第三阶段先不引入复杂中间件，重点补齐真实项目落地时最容易被忽略的基础能力：接口数据是否来自缓存、是否发生降级、数据生成时间是否可追踪、前端是否能明确展示数据状态。

本阶段保持轻量实现，服务于演示、联调和后续生产化扩展。

## 已实现范围

### 1. 分析看板响应可观测

`GET /api/dashboard` 和 `POST /api/dashboard/refresh` 现在同时在响应头和响应体透出运行状态。

响应体新增字段：

| 字段 | 说明 |
| --- | --- |
| `generatedAt` | 数据生成时间，由分析服务生成 |
| `rangeDays` | 当前统计周期天数 |
| `cacheStatus` | 缓存状态，支持 `HIT`、`MISS`、`STALE` |
| `stale` | 是否使用最近一次成功数据进行降级返回 |

响应头：

| Header | 说明 |
| --- | --- |
| `X-Cache-Status` | 缓存状态 |
| `X-Data-Stale` | 是否为降级数据 |
| `X-Data-Generated-At` | 数据生成时间 |
| `X-Data-Range-Days` | 统计周期天数 |

### 2. 运营端展示数据健康状态

小程序首页新增轻量状态条，但仅对 `STAFF`、`ADMIN` 等内部运营角色展示。普通用户首页不展示缓存、降级、生成时间等技术状态，避免干扰业务使用。

内部角色可看到：

- 数据来源状态：缓存命中、实时生成、降级数据、演示数据。
- 当前统计周期。
- 数据生成时间。
- 降级时给出明确提示，避免运营人员误判数据新鲜度。

### 3. 测试契约

分析服务控制器测试已覆盖：

- 缓存状态响应头。
- 降级状态响应头。
- 响应体中的 `cacheStatus` 和 `stale` 字段。
- 服务不可用时返回 503 业务错误码。

## 当前边界

本阶段暂不做以下内容：

- 不引入 RabbitMQ、Kafka 等消息队列。
- 不接入 Prometheus、Grafana、ELK 等完整监控链路。
- 不做复杂分布式追踪。
- 不做自动告警平台。

这些能力更适合在接口、流程和数据闭环稳定后进入第四阶段。

## 验收标准

1. 分析看板接口正常返回业务数据。
2. 响应体包含 `cacheStatus`、`stale`、`generatedAt`、`rangeDays`。
3. 小程序普通用户首页不展示数据状态条。
4. 小程序运营和管理员角色能展示数据状态条。
5. 当分析服务使用最近一次成功缓存时，内部前端能明确看到降级状态。
6. 后端测试通过。
